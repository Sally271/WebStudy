Git?

Linus Torvalds(이 업계의 Gordon Ramsay)가 만든
Distributed Version Control System(분산 버전 관리 시스템).
대충 분산(사용자들 각각에(local))해서 버전관리 하게 해주는 프로그램...
사용자는 로컬 환경으로 소스를 clone(싹 다 복사해옴)해서 편집하고,
다시 업로드. 구글 드라이브의 버전 관리랑 다를게 없..지만
수정 여부를 알려주는게 편-안하다.

Git Bash?

Git은 기본적으로 CLI(Command Line Interface)를 통해 사용.
Git Bash는 Windows의 cmd, Linux와 Mac의 terminal과 같은 역할을 한다.
이들 운영체제마다 명령어가 다르다는 문제점을 Git Bash가 극복한 것.
Git Bash로는 Windows에서도 Linux의 명령어를 쓸 수 있다.

Repository?

Repository(Repo)는 Remote Server 상에서 구분되는 프로젝트 단위이다.
즉, 로컬 저장소와 구분되는 원격 저장소.
클라우드 드라이브의 디렉토리쯤 된다고 생각하면 되겠다.
Repository를 clone받으려면 해당 Repository의 URL이 필요한데,
Git Repository는 URL 맨 끝에 .git 확장자가 붙는 식으로 표현된다.

Remote Server(Remote)?

Remote Server(Remote)는 클라우드 서버... 느낌이다.
이 서버를 제공하는 대표적인 업체가 Github임.
Git을 사용할 때는 어떤(어느) Remote Server에 변경사항을 업로드할지 정해야 하는데,
반드시 하나의 Remote만 사용할 수 있는 것은 아니므로
내가 사용할 Remote의 이름을 정해줘야 한다.
이때 주로 사용하는 관례적인 이름이 'Origin'이다.
Remote와 Origin을 혼용하기도 한다.

Root Folder?

Web상의 Git과 연결된 local folder를 말함.
우선 local에 폴더를 하나 만들고 우클릭 -> Git Bash로 실행

Root Folder 연결 command:

git init
"초기화 작업" = "Git 저장소의 생성"
local폴더 내에 .git 폴더를 만듬. 이는 기존 운영체제의 일반폴더가 깃에서 관리하는 저장소로 바뀜을 뜻함
폴더 내용물을 수정한다면, 그 기록이 객체 형태로 .git 폴더에 저장되는 것.
git이라는 실행어와 init이라는 명령어로 구분되는것.
현재 폴더를 초기화하려면 git init . 이라 치면 된다. (. 은 current dir라는 뜻)
initialization이 성공적이라면 터미널 맨 오른쪽에 (master)라고 뜨게 된다.

git remote add origin repository address
여기서 repository address는 Github의 Repository 페이지에서 복사해온다.
git remote 는 원격 저장소를 등록하는 git 명령어이다.
git remote -v 를 하면 등록된 원격 저장소를 확인할 수 있다.

Pull & Push 과정

Push 전엔 항상 Pull이 있다.

git pull origin master
지정한 Repository에서 모든 자료를 Root Folder로 끌어오는 것

git status
local folder와 Git과의 sync check.
깃은 "상태"(status)라는 개념이 있음. 이는 효율적인 코드 이력의 관리를 위함.
Git Bash에다 쳐보면 "use "git add <file>..." to include in what will be committed" 라고 뜸
참고로 <는 &lt; (less than)라고 쳐야 표시된다. >는 &gt; (greater than). 그럼 &는? &amp;

git add .
git status로 확인한 모든 변경된 파일들을 tracking해달라는 뜻. (스페이스와 .까지 명령어에 포함됨)

git commit -m "message"
commit함과 동시에 해당 commit에 대한 메세지를 남김. (따옴표까지 명령어에 포함됨)
commit 단계에서
Please tell me who you are .Run
git config --global user.email "you@example.com"
git config --global user.name "Your Name" to set your account's default identity.
이런 에러 메세지가 뜬다면, 에러메세지가 말해주는대로
$ git config --global user.name "user name"
$ git config --global user.email "user e-mail"
이라고 적어주면 된다.
이후 git log를 통해 commit 여부를 확인할 수 있다.

git push origin master
실제로 Git에 push하는 것.
master 앞에 +라고 쓰기도 하던데 뭔지 모르겠음.

일단 push는 크게 세 단계로 이뤄지는듯. add -> commit -> push
처음엔 내가 편집하는 파일들이 들어있는 워킹 디렉터리가 있고 (이곳의 파일들은 untracked)
add를 하면 "staging", 이때 등록이 됨(tracked)(즉, git add . 을 하면 모든 파일을 "Staging Area에 넣는다")
이때 add를 취소하려면 git reset --mixed HEAD -> commit을 취소하고 파일들은 unstaged 상태로 워킹 디렉토리에 보존.
여기서 -- 다음에 오는건 soft, mixed, hard가 있는데 mixed가 기본이고(즉, -- 없이 그냥 reset HEAD 하면 mixed로 됨)
soft는 commit만 취소, 파일들은 staged로 보존,
hard는 commit 취소와 함께 working directory의 파일들까지 삭제.
즉, hard는 Remote에 있는 마지막 commit 이후의 working dir.와 add했던 파일들이 모두 사라지므로 주의!

강의 들으면서 깨달은 것들

깃은 코드의 이력을 관리하는 것임. 그래서 '상태'라는 개념 존재. 보다 효율적으로 코드 이력을 관리하기 위함.
상태는 세 개가 있음. "워킹 디렉터리", "스테이지", ".git 저장소"
직접 워킹 디렉토리에서 저장소로 저장하는것은 불가능하므로(파일 수 많아지면 일일이 하기 어려움)
중간에 스테이지 영역이라는 상태를 둠.
워킹디렉토리의 파일을 스테이지영역에 등록. -> 관리한다는 것을 인지하게 됨 (tracked)
따라서 한 번 등록한 파일은 modified라고 뜨고(changes not staged for commit), 등록 안 된 파일은 untracked라고 뜸.
스테이지에 등록한 파일을 .git에 저장하게 되는데 이를 기록=커밋이라 함.
staging 이후 파일을 수정하면 unstage로 자동으로 바낌.(워킹디렉토리로 돌아감)
